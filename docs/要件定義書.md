# 要件定義書：保々中学校同窓会 クイズ大会システム

## 1. システム構成・アーキテクチャ
* **フロントエンド（参加者用・投影用・管理者用）**: Next.js (React)
  * レスポンシブ対応およびSPA/SSRを活用し、高速な画面レンダリングを実現。
* **バックエンド・API**: FastAPI (Python)
  * 高速なAPIレスポンス、並行処理に強いASGIフレームワークとして採用。
* **データベース・ORM**: PostgreSQL / Prisma
  * リレーショナルデータ（チーム情報、問題、解答ログ、スコア等）を型安全に管理。
* **リアルタイム通信**: WebSockets (FastAPI上で実装、あるいはPusher/Socket.ioなどのマネージドサービスを検討)
  * 管理者の画面操作（問題切り替えなど）を参加者端末および投影画面へミリ秒単位で同期するため必須。

## 2. 機能要件

### 2.1 参加者画面（モバイル端末向け）
* **ログイン・待機機能**: 割り当てられたチーム名・パスコード等を選択・入力し、出題待ち状態（待機画面）となる。
* **クイズ解答機能**:
  * **問題受取**: 管理者が「出題」操作を行うと、画面が自動で切り替わり問題文および選択肢が表示される。
  * **ベッティング選択**: 「x1 / x2 / x3」のいずれかの倍率を画面上で選択する。
  * **解答送信**: 選択肢をタップしサーバーに送信する。送信時のタイムスタンプを付与（あるいはサーバー到達時刻で判定）し、速度評価に用いる。
* **結果フィードバック確認機能**: 締め切り後、自分のチームの正解/不正解、獲得ポイント、現在の総合スコアが表示される。

### 2.2 管理者画面（操作用PC・タブレット向け）
* **ダッシュボード・状態管理**: 
  * 全8チームの接続状態（オンライン/オフライン）、および現在出題中の問題の各チームの解答提出状況を一覧表示。
* **進行コントロールパネル**:
  * 「問題表示」「タイマースタート」「締め切り」「正解発表」「ランキング表示」「次の問題へ」などのステート遷移ボタン。
* **演出・BGMコントロール**:
  * フロントエンド側へBGM再生/停止/効果音のトリガーとなるイベントを発火させる機能。

### 2.3 投影画面（スクリーン・PC向け）
* **ステート別大画面表示**:
  * **問題提示中**: 問題文、選択肢、および各チームの解答完了状況アイコン。
  * **正解発表中**: 正解の選択肢のハイライトアニメーション。
  * **ランキングボード**: 現在のスコアに基づく、順位変動のアニメーションを伴う順位表。
  * **マジョリティ結果**: 会場全体の得票率を可視化するグラフ表示（円グラフ・棒グラフ）。

### 2.4 クイズ・ポイント計算ロジック
* **基本の加減点ロジック**: 
  * 正解：基本スコア × 選択したベッティング倍率（x1, x2, x3）
  * 不正解：マイナス（基本スコア × 選択したベッティング倍率）
* **解答速度ボーナス（減点タイマー方式）**:
  * 出題・タイマースタートからの経過時間をミリ秒単位で計測。
  * 早く解答したチームほどポイントが高くなるよう、時間の経過とともに減少するボーナスポイントを計算し、基本得点等に加算する。
* **マジョリティ・ボーナス（価値観同調ゲーム）**:
  * 正解を持たないアンケート形式の問題。
  * 集計後、最も投票の多かった選択肢（マジョリティ）を選んだチームにポイントを付与。
  * 獲得ポイントは、その選択肢の会場全体での「得票率（％）」に比例・連動するロジックとする。
* **各賞データ抽出機能（バックエンド）**:
  * システムの全解答・ベットログ、タイムスタンプを解析し、「マジョリティ一致率が極端に低い」「ベット成功率・最速解答頻度が最も高い」などの条件でデータを抽出し、結果を返すエンドポイント。

### 2.5 戦績リザルト・事後共有機能
* イベント終了時に、システムが各チーム固有の戦績ページURLを生成。
* 同URLにアクセスするためのQRコードをプロジェクター投影画面、および各デバイス画面に表示。
* 参加者は自身のスマホでQRを読み取ることで、大会を通したクイズ成績詳細（正答率、平均速度、ベットログなど）を振り返ることができる。

## 3. 非機能要件

### 3.1 リアルタイム性・パフォーマンス
* 全8チームと管理者、投影用端末（計10台程度）からのWebSocket接続を維持し、管理者の操作（Next）から参加者画面への反映遅延を抑える（目標300ms以内）。
* 解答の早押し判定における不公平感をなくすため、フロントエンドとバックエンドの時刻ズレや通信ラグに対する考慮（必要ならNTP同期的な処理や、サーバー到達時計測を統一）を行う。

### 3.2 可用性・障害耐性
* 万が一、イベント進行中にブラウザクラッシュやスマホの誤操作での戻るボタン、再読み込み、一時的な通信切断が発生しても、ページをリロードした瞬間にサーバーから現在の正しいステートメント（例：第3問解答中）を取得し、即座に復帰できる「ステートレス・サーバー主導」の設計を行う。

### 3.3 UI / UX
* 祝いの席（同窓会）にふさわしい、派手でわかりやすいフィードバック（正解・不正解時のアニメーション効果、解答受け付け時のサウンド連携等）。
* スマホ画面では、押し間違いを防ぐためボタンサイズやマージンに留意し、ストレスの無い操作性を提供する。

### 3.4 ホスティング・インフラストラクチャー
* 想定最大同時アクセス数は管理用含め数十セッション程度のため、大規模構成は不要。
* Next.jsフロントエンド: Vercel 等のエッジネットワーク利用推奨。
* FastAPIバックエンド・WebSocketサーバ: Render.com、Railway、AWS App Runner など、WebSocketsに対応したPaaS環境。
* データベース: Supabase、Neon などのフルマネージドPostgreSQL。
